## Features

- Supabase Auth (email/password) via `@supabase/ssr`
- Server-side session refresh in `hooks.server.ts` (secure httpOnly cookies)
- `event.locals.user` / `event.locals.session` available in every load/action
- Protected `/dashboard` route (server `load` redirects unauthenticated users)
- Sign up / log in / sign out flows with SvelteKit form actions + `enhance`
- `@sveltejs/adapter-node` for containerized deploy (Render / Fly / any OCI host)
- `.env.example` documenting every required variable

## Quick start

```bash

```

## Supabase setup

1. Create a project at [supabase.com](https://supabase.com).
2. In **Project Settings → API**, copy:
   - `Project URL` → `PUBLIC_SUPABASE_URL`
   - `anon public` key → `PUBLIC_SUPABASE_ANON_KEY`
3. (Optional) Under **Authentication → Providers**, enable email confirmations.
   With confirmations on, new users receive a magic link before they can sign in.

> Note: SvelteKit only exposes env vars prefixed with `PUBLIC_` to the client.
> The anon key is safe to expose — Row Level Security is what protects your data.

## How auth works here

- `src/hooks.server.ts` — creates the Supabase server client on every request,
  refreshes the session, and writes `set-cookie` headers onto the response.
  It also populates `event.locals.user` / `event.locals.session`.
- `src/lib/supabase/client.ts` — browser client for client-side code.
- Form actions in `login` / `signup` / `dashboard` call
  `event.locals.supabase.auth.*` directly on the server.

This is the official Supabase-recommended `@supabase/ssr` SvelteKit pattern.
