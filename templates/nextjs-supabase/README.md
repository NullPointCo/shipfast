## Features

- Next.js 14 App Router + TypeScript
- Supabase Auth (email/password) via `@supabase/ssr`
- Session refresh in `middleware.ts` (secure httpOnly cookies)
- Server client (`lib/supabase/server.ts`) for Server Components & Actions
- Browser client (`lib/supabase/client.ts`) for Client Components
- Protected `/dashboard` route (server check redirects unauthenticated users)
- Sign up / log in / sign out flows with Server Actions + `useFormState`
- `.env.example` documenting every required variable
- Dockerfile for containerized deploy (Render / Fly / any OCI host)

## Quick start

```bash
npm install
cp .env.example .env.local   # fill in your Supabase URL + anon key
npm run dev                  # http://localhost:3000
```

## Supabase setup

1. Create a project at [supabase.com](https://supabase.com).
2. In **Project Settings → API**, copy:
   - `Project URL` → `NEXT_PUBLIC_SUPABASE_URL`
   - `anon public` key → `NEXT_PUBLIC_SUPABASE_ANON_KEY`
3. (Optional) Under **Authentication → Providers**, enable email confirmations.
   With confirmations on, new users receive a magic link before they can sign in.

> The anon key is safe to expose in the browser — Row Level Security is what
> protects your data.

## How auth works here

- `middleware.ts` — refreshes the session on every request and writes rotated
  cookies back onto the response, keeping SSR and the browser in sync.
- `lib/supabase/server.ts` — Supabase client for Server Components, Route
  Handlers, and Server Actions, backed by Next.js's cookie store.
- `lib/supabase/client.ts` — browser client for Client Components.
- `/dashboard` calls `supabase.auth.getUser()` in a Server Component and
  `redirect("/login")` if there is no session.

This is the official Supabase-recommended `@supabase/ssr` Next.js pattern.
